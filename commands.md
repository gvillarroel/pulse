# pulse commands

This document defines the CLI contract for `pulse`.

As of 2026-04-02, the implemented command set is intentionally reduced to three primary commands:

- one command to determine which repositories should be processed
- one command to process that list and update a reusable state directory
- one command to render an interactive report from a reusable state directory

The tool should not rely on implicit internal sets or a hidden global catalog. Everything important should come from explicit parameters or explicit files.

The preferred explicit file format is YAML, with optional references to CSV inputs.

## Current Implementation Status

Implemented today:

- `pulse list`
- `pulse run`
- `pulse report`
- CSV and YAML repository intake
- persistent state under `--state-dir`
- focus pattern configuration
- optional weekly history aggregation
- self-contained HTML reporting

Deferred from this command contract:

- provider discovery flags such as `--provider`, `--owner`, or `--pushed-since`
- direct provider-backed listing from `pulse list`
- direct provider-backed execution from `pulse run`
- dedicated query/export subcommands beyond `report`

## Principles

- few commands
- reproducible execution
- explicit state on disk
- automatic resume from the specified state directory
- easy piping between inputs and outputs

## General Conventions

Commands should accept these options where relevant:

- `--json` for structured output
- `--state-dir <path>` to define where incremental state lives
- `--workspace <path>` to define where temporary execution work happens
- `--concurrency <n>` to control parallelism
- `--progress` to show execution progress

`--state-dir` is the local source of truth for:

- previously fetched repositories
- checkpoints
- derived metadata
- partial and final reusable results

`--workspace` is operational space for:

- temporary clones or workdirs when needed
- ephemeral execution files
- artifacts that do not need to survive as durable state

## Command 1: Resolve Repositories

### `pulse list`

Resolves the repositories that should be processed.

This command does not analyze repositories. Its job is to produce an explicit, reproducible target list.

Currently supported sources:

- a YAML config file
- an input CSV
- a previously generated list

Examples:

```bash
pulse list --config pulse.yaml
pulse list --input repos.csv
pulse list --input repos.csv --out targets.csv --format csv
pulse list --config pulse.yaml --format json
```

Implemented flags:

- `--config <yaml>`
- `--input <csv>`
- `--out <path>`
- `--format <text|csv|json>`

Command contract:

- normalizes repository identifiers
- removes duplicates
- emits one final explicit list
- can write that list to a file for `pulse run`
- if a YAML config is provided, it resolves repositories from the referenced CSV path

Planned but not yet implemented:

- provider filters in the CLI
- discovery-backed target generation without an explicit CSV

Suggested minimum output columns:

- `repo`
- `provider`
- `owner`
- `owner_color`
- `team`
- `team_color`
- `url`

## Command 2: Process Repositories

### `pulse run`

Processes a repository list and updates a reusable state directory.

This is the main command of the system. It should:

- read the target list
- clone or update repositories
- compute baseline metrics
- apply deeper analysis to configured focus paths
- compute historical evolution
- save checkpoints and reusable results
- resume automatically when prior work exists in `--state-dir`

Examples:

```bash
pulse run --config pulse.yaml --state-dir ./state --workspace ./work
pulse run --input targets.csv --state-dir ./state --workspace ./work
pulse run --input repos.csv --state-dir ./state --workspace ./work --concurrency 16
pulse run --input targets.csv --state-dir ./state --workspace ./work --with-history
pulse run --input targets.csv --state-dir ./state --workspace ./work --focus "docs/**/*.md"
```

Accepted inputs:

- a YAML config file
- a CSV generated by `pulse list`
- a raw input CSV

Implemented flags:

- `--config <yaml>`
- `--input <csv>`
- `--state-dir <path>`
- `--workspace <path>`
- `--concurrency <n>`
- `--with-history`
- `--history-window <duration>`
- `--focus <pattern>`
- `--focus-file <path>`
- `--fail-fast`

Command contract:

- if a repository was already fetched, reuse the local Git data
- if a stage was already computed and is still valid, do not recompute it
- perform baseline analysis across the repository and deeper analysis only for configured matching paths
- if execution is interrupted, the next run continues from checkpoints
- if some repositories fail, the rest continue by default
- leave `--state-dir` ready for another run next week

Accepted today but not yet functionally active:

- `--workspace`
- `--concurrency`
- `--history-window`

Planned but not yet implemented:

- provider discovery flags on `run`
- semantic analyzer toggles such as `--with-rust-analyzer`

Expected contents of `--state-dir`:

- local Git repository cache
- per-repository and per-stage checkpoints
- repository metric snapshots
- file metric snapshots
- weekly historical aggregates
- enough execution metadata to resume safely
- persisted focus-pattern configuration used for the run

## Focused Analysis Rules

The configuration model should support targeted deeper analysis for selected files and folders.

Expected pattern capabilities:

- all files under a specific folder
- files with a specific extension under a specific folder
- files with a specific extension anywhere in the repository
- exact file paths
- include and exclude glob rules

Examples:

- `src/data/**`
- `src/data/**/*.csv`
- `**/*.csv`
- `docs/reference/*.md`
- `Cargo.toml`

## Expected Workflow

### Option A: Explicit two-step workflow

```bash
pulse list --config pulse.yaml --out targets.csv
pulse run --config pulse.yaml --input targets.csv --state-dir ./state --workspace ./work
```

Option A is the current production workflow when the repository list should be inspected or versioned before processing.

### Option B: YAML-driven one-step workflow

```bash
pulse run --config pulse.yaml --state-dir ./state --workspace ./work
```

Option B is the shortest current path when the YAML already points to the repository CSV and carries the shared analysis settings.

## V1 Scope

For the first version, the commands only need to define support for:

- determining the target repositories
- concurrent processing
- incremental storage in an explicit directory
- resume behavior
- reuse of previous executions

## Command 3: Render Report

### `pulse report`

Renders a self-contained interactive HTML report from an existing `--state-dir`.

This command does not fetch or analyze repositories again. Its job is to turn the persisted SQLite state into a shareable artifact with multiple navigable sections inside a single HTML file.

Examples:

```bash
pulse report --state-dir ./state
pulse report --state-dir ./state --out ./reports/team-overview.html
pulse report --state-dir ./state --title "My Company Engineering Portfolio"
pulse report --config pulse.yaml --state-dir ./state
```

When `--config` is provided, it may still supply presentation options, but it must not trigger fresh repository scanning or redefine metrics at render time.

Example:

```yaml
report:
  ai_docs:
    include:
      - AGENTS.md
      - .sdlc/architect.md
    exclude:
      - archive/**
```

AI-document and similar derived signals must already have been computed by `pulse run` and persisted into the state directory before rendering.

Expected flags:

- `--state-dir <path>`
- `--out <html>`
- `--title <text>`

Command contract:

- reads only the persisted state under `--state-dir`
- writes one HTML file that works without a server
- exposes multiple dimensions such as overview, languages, repositories, history, and failures
- keeps charts interactive so the operator can click through dimensions inside the same file
- defaults to writing under `--state-dir/exports/report.html` when `--out` is not provided
- does not walk repositories, read Git blobs, or execute new history scans during rendering
